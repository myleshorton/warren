warren

A network layer with no center — move any content between peers, opaquely.

Warren is a serverless P2P substrate: blinded discovery, hole-punched direct connections, signed append-only feeds, content-addressed blobs, and verified swarm sync. It moves opaque, verifiable bytes between peers with nothing in the middle — and it doesn't care what the bytes are.

› no infrastructure in the data path › content-agnostic: video · chat · files · feeds › discovery a censor can't enumerate
the substrate

A serverless data layer, not an app

Warren gives you the hard parts of a peer-to-peer system as small, verifiable Rust crates. What you send over it — a video, a message, a file, a signed data feed — is your payload, not Warren's concern.

01

No center

Peers connect directly — hole-punched over a single socket. The DHT is used only for rendezvous; bulk data never touches a server, gateway, or required relay.

02

Opaque by construction

Discovery runs under blinded, rotating topics. Content is addressed by the hash of its ciphertext. There's nothing stable for a censor to enumerate or fingerprint.

03

Content-agnostic

The core abstractions — a signed append-only log and a content-addressed blob store — are general. Video is one payload; chat, files, and data feeds are others.

driverconnectivity

A DHT node plus single-socket NAT hole-punching. Dials a peer by id, punches a direct path (incl. birthday-punch for symmetric NAT), and hands back a raw channel.

swarmdiscovery

Kademlia routing: announce under a topic, look up who else did. Turns a shared key into a set of reachable peers.

transfersync

Verified feed + blob sync with multi-source swarming — rarest-first, work-stealing, resumable, streaming, every chunk checked as it lands.

feedlog

A signed, append-only, per-block-verifiable log (O(log n) incremental Merkle head). Content identity, decoupled from network identity.

blobstorage

A content-addressed chunk store. Split a payload, name each piece by its hash, reassemble anywhere from any source that holds the pieces.

cryptoprimitives

Ed25519 signatures, BLAKE3 hashing, and an XChaCha20 sealing envelope — seekable per-item encryption that keeps streaming progressive while blobs stay opaque.

portmapreachability

PCP (RFC 6887) + UPnP-IGD port mapping and reflexive address discovery, so a peer can advertise a path others can actually reach.

wire protocol

How two peers actually meet

There is no signaling server. The same DHT that finds content also brokers the connection — peers trade candidates over it, then punch a direct path — which a Noise handshake then turns into a forward-secret, mutually-authenticated channel bound to the peer's node id. What's left honest (see the limits): the discovery-layer signaling isn't signed yet, and the wire carries no traffic obfuscation.

1 · signalrendezvous

The target announces under its own id; any DHT node holding that record becomes a coordinator, and a Signal relays each side's candidates + NAT type through it. No dedicated signaling service — unlike WebRTC, rendezvous is the DHT.

2 · candidatesreachability

Each peer offers a few addresses — local, reflexive (a STUN-like probe to nearby peers), and an optional PCP/UPnP port-map — most-routable first. The coordinator sees the control address; it can't observe the data-socket candidates, so the endpoints fill those in.

3 · punchnat traversal

A one-byte PROBE / ACK opens a direct UDP path. For one-sided symmetric NAT it leans on the birthday paradox (panel below). Two peers both behind symmetric NAT aren't punched — no relay sits in the path, by design.

4 · channeltransport

The channel runs a Noise XX handshake — forward-secret, mutually authenticated, and bound to the peer's node id — so the selective-repeat framing is AEAD-encrypted end to end. Every datagram carries its own nonce behind a sliding replay window, so forged, tampered, or replayed packets are dropped as loss rather than breaking the stream. Clips are additionally sealed at the content layer (XChaCha20, named + signed by ciphertext hash) so blind mirrors can carry bytes they can't read.

birthday punch
For one-sided symmetric NAT, the random-port side opens 256 sockets while the other sprays 1750 probes across ~64k ports — no agreement on which ports, just enough of both to collide. A probe landing on an open socket wins: 1 − (1 − 256/64512)^1750 ≈ 99.9%.

Group chat / messaging

merge built

Rooms map onto merged member feeds (text as records, media as blobs). Live-tail replication is built — subscribe and get pushed new blocks in real time, each verified against the signed head, no polling. Swarm-failover subscription and blind-mirror store-and-forward are built too: a feed is found by topic and tailed from any provider, and a mirror keeps a verified replica serving so the feed stays live even after its author goes offline — the mirror is never trusted, since every block still verifies against the author's key. And the multi-writer causal merge is built (Autobase-style): each message carries a version-vector clock, and a deterministic linearizer merges many writers' feeds into the one causal order every participant computes identically — proven to converge over a real DHT. What's left is the app wiring — a chat client driving the room view.

the channel

A channel is three gates, not one key

"A shared key" is only the first gate — the one that controls who can find and read. A Warren channel stacks two more independent gates on top, each enforced by different cryptography and each holding against a different adversary. The upshot: seizing the key does not hand a censor the room.

1 · psk
Discovery & read. The rotating topic is H(domain ‖ psk ‖ epoch) — every holder derives the same handle, and to anyone else it's noise that changes each epoch.
who can find
2 · content key
Decryption. A separate key derives the KEK that unwraps each blob's random per-item key. Withhold it and a peer becomes a blind node — it serves ciphertext it can never open.
who can decrypt
3 · roster
Membership & authorship. Signed member.add / member.remove records fold, in causal order, from a founder genesis — a change counts only if its author was already a member (transitive vouching). Holding the PSK does not make you one.
who can post
the pskread gate

A freely-shareable capability: possession lets you compute the topic and decrypt reads. Because it's shareable it can leak — so Warren never lets the read gate be the only gate.

the content keydecrypt gate

Splitting discovery from decryption lets an untrusted volunteer host a channel's bytes without being able to read them — availability without trust. The mirror carries ciphertext plus the signed feed, verifies every block, and opens none.

the rosterwrite gate

The write-and-identity defense against a censor who has the PSK. Membership is a signed, append-only log folded from the founder; a leaked key buys reading, not the ability to post, flood, or impersonate a member.

threat posture

What makes it hard to censor

Warren treats opacity to a censor as the first requirement, not a feature bolted on later. Several independent properties do the work — and they hold whatever the payload is. The aim is to raise the cost of blocking, enumerating, and de-anonymizing a swarm, not to promise perfect anonymity against a global observer who can watch every link at once — that adversary is out of scope today (see the limits below).

Blinded, rotating discovery

The discovery topic is hash(psk, epoch) — meaningless without the key and different every epoch. No fixed handle to add to a blocklist.

Decoupled identities

A peer's node id (where it is) is independent from its content id (who authored). Hosting and authoring don't collapse to one stable name.

Per-epoch re-announce

Membership refreshes on a clock (keep_announced), so the swarm heals through churn without a registry anyone could subpoena.

Ciphertext addressing

Blobs are named by the hash of their ciphertext under a random key — so a censor holding a banned item can't compute its id to hunt hosts. Content-addressing, minus the fingerprint.

Nothing in the data path

Optional volunteer blind peers reseed encrypted bytes they can't read. Nothing is required, nothing is central — availability without infrastructure.

Vouching, not the key, grants entry

A leaked PSK lets an intruder read — but not join or post. Membership is a signed roster folded from a founder, and only a vouched-in member authors records others accept. The secret is a capability, not a seat: seizing it doesn't let a censor flood, impersonate, or take over the room.

No single door to bolt shut

An invite is self-contained — it carries its own live bootstrap peers, so there's no one foreign address every newcomer must reach, and invites travel socially rather than through a directory. Richer rendezvous that degrades under active blocking (cover-DHT, domestic/local fallbacks) is designed, not yet built.

What Warren borrows

The signed append-only feed echoes Hypercore. The per-chunk BLAKE3 verified streaming echoes iroh's bao. The hole-punching uses the same NAT tricks the whole field shares. Standing on shoulders, deliberately.

What they do better

IPFS: a vast ecosystem and durable persistence. iroh: QUIC polish, throughput, and batteries-included protocols. Holepunch: a mature runtime shipping real apps like Keet. Warren is a research prototype by comparison.

honest limitations

What doesn't work yet

A serverless design pays for its properties. These are framework-level costs — the app layer adds its own (a mobile client is suspended in the background; a desktop peer isn't).

nat

Two symmetric-NAT peers may not punch. With no relay in the path, a pair behind carrier-grade symmetric NAT can fail to connect directly. Warren leaves that pair unconnected by design rather than routing data through infrastructure; a reachable peer or a volunteer blind peer bridges availability.

real-time

No chat client yet. The real-time substrate is complete — live-tail push, swarm-failover subscription, blind-mirror store-and-forward, and deterministic multi-writer causal merge all work and are tested over a real DHT. What's missing is an application on top: a chat client that drives the room view and publishes with a causal clock. The framework is ready for one; none is built.

signaling auth

The channel is encrypted and identity-bound now; the DHT signaling isn't signed yet. A Noise handshake means you only ever connect to the exact node id you dialed, over a forward-secret AEAD channel — but the discovery-layer announce/signal records are still unsigned, so a peer could announce under another's id or hand a dialer a forged punch target. Signed announces + capability tokens are the next step.

transport

No obfuscated transport yet. Discovery is blinded and identities decoupled, but deep packet inspection can still see that a peer speaks a P2P protocol — just not what, or with whom. A pluggable obfuscated transport is planned.

rendezvous

Bootstrap can still be blocked. An invite carries its own live bootstrap peers, so there's no single fixed foreign address to censor — but if every peer in a given invite is unreachable, there's no fallback rendezvous yet. Cover-DHT and domestic/local rendezvous that degrade gracefully under active blocking are designed, not built.
